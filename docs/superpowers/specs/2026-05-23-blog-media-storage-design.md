# Blog Media Storage Slice Design

Status: draft for review
Date: 2026-05-23
Source PRD: `docs/_personal_blog_PRD_lichnyj_sajt_blog_free_first_NextJS_PayloadCMS_MongoDB.md`
Builds on: `docs/superpowers/specs/2026-05-22-blog-foundation-design.md`

## Goal

Make blog media uploads production-safe for Vercel by moving Payload `Media` uploads from local filesystem storage to Vercel Blob, then render post cover images on public blog pages.

This slice proves the media path:

Admin uploads image in Payload -> Payload stores file in Vercel Blob -> Media document stores public URL/metadata -> post uses image as `coverImage` -> `/blog` and `/blog/[slug]` render the image.

## Scope

Included:

- Add the Payload Vercel Blob storage adapter.
- Configure the existing `media` collection to use Vercel Blob when `BLOB_READ_WRITE_TOKEN` is available.
- Fail safely on Vercel if Blob storage is not configured, so production does not silently write uploads to ephemeral local filesystem.
- Keep local development usable without Blob credentials when only CMS/blog behavior is being developed.
- Restrict media uploads to web image MIME types for this slice.
- Add image sizes/thumbnail settings for admin and public rendering.
- Render `coverImage` on `/blog` cards and `/blog/[slug]` pages.
- Add Open Graph image metadata for post pages when a cover image exists.
- Configure `next/image` remote image allowlisting for the Vercel Blob public hostname.
- Update `.env.example` and README with Blob variables and setup notes.
- Remove the credential-like stray line currently present in `.env.example`.

Excluded:

- Rich text embedded image nodes.
- Manual custom upload API routes.
- Private Blob files.
- Image deletion/reconciliation jobs.
- Bulk migration of existing local media files to Blob.
- Image editing/cropping UI beyond Payload's built-in upload/admin behavior.
- Comments, projects, analytics, RSS, and wider visual redesign.

## Current Project Context

The foundation slice is implemented:

- `payload.config.ts` already uses Payload v3.84.1 with MongoDB and Lexical.
- `collections/Media.ts` currently uses `upload: true`.
- `collections/Posts.ts` already has `coverImage` as an upload relationship to `media`.
- Public post queries use `depth: 2`, so `coverImage` can resolve as a `Media` object.
- Public routes live in `app/(site)`.
- `next.config.ts` is wrapped with `withPayload`.
- `.env.example` currently includes required foundation env vars, but also contains an extra credential-like line that must be removed.

## Architecture

Use Payload's official Vercel Blob storage adapter, not a custom upload route.

- `@payloadcms/storage-vercel-blob` owns writing uploads to Blob.
- The existing `Media` collection remains the single content model for images.
- Payload stores standard upload metadata such as `url`, `filename`, `mimeType`, `filesize`, `width`, and `height`.
- Public UI reads the resolved `coverImage` object from the existing post queries.
- A small `MediaImage` component converts a Payload `Media` object into a safe `next/image` render.

The adapter should be enabled only when a token exists, except on Vercel where missing Blob configuration is a deployment error.

Rationale:

- Local development can still run without a Vercel account/token.
- Production cannot accidentally rely on local upload storage, which is unsafe on Vercel.

## Environment Variables

Required for production media uploads:

- `BLOB_READ_WRITE_TOKEN`: server-side Vercel Blob read/write token.
- `NEXT_PUBLIC_BLOB_HOSTNAME`: exact public Blob hostname, for example `<store-id>.public.blob.vercel-storage.com`.

Existing variables remain:

- `DATABASE_URI`
- `PAYLOAD_SECRET`
- `NEXT_PUBLIC_SITE_URL`

`NEXT_PUBLIC_BLOB_HOSTNAME` is not secret. It exists to keep `next/image` `remotePatterns` strict instead of allowing all Blob hostnames.

## Media Collection

The existing `Media` collection should stay small:

- `alt`: required text.
- Payload upload metadata fields generated by the upload feature.

Upload settings:

- Allow only image MIME types needed for the public blog:
  - `image/jpeg`
  - `image/png`
  - `image/webp`
  - `image/gif`
- Generate a small admin/public thumbnail.
- Generate a post cover size suitable for blog cards and post hero images.
- Preserve focal point support if Payload config supports it cleanly.

Do not add custom `blobUrl` or `storageKey` fields in this slice unless the installed adapter does not expose enough metadata through Payload's standard upload fields. The public source of truth should be `media.url`.

## Public Rendering

### Blog Index

`/blog` should render cover images for posts that have a resolved `coverImage`.

Acceptance:

- Posts without cover images still render cleanly.
- Cards do not shift layout unexpectedly.
- Images use `alt` from the Media document.

### Blog Post

`/blog/[slug]` should render the cover image near the top of the article.

Acceptance:

- Published post with cover image shows the image.
- Published post without cover image still renders.
- Image dimensions or a fixed aspect-ratio wrapper prevent layout shift.

### Metadata

Post metadata should include Open Graph images when `coverImage.url` exists.

Acceptance:

- `generateMetadata` falls back to title/excerpt when there is no image.
- It does not throw when `coverImage` is a string ID instead of a populated object.

## Next Image Configuration

Configure `next.config.ts` with a strict remote pattern:

- protocol: `https`
- hostname: `process.env.NEXT_PUBLIC_BLOB_HOSTNAME`
- pathname: `/**`

If the hostname is missing locally, the app can still build, but remote Blob images will require the env var before rendering through `next/image`.

## Error Handling

- If `BLOB_READ_WRITE_TOKEN` is missing on Vercel, fail during server config initialization with a clear message.
- If `NEXT_PUBLIC_BLOB_HOSTNAME` is missing, log/document that optimized remote image rendering is not configured; production verification should treat this as incomplete.
- If an upload has no public `url`, public components should skip the image rather than throw.
- If image metadata is incomplete, use a stable aspect-ratio wrapper with `fill`.

## Testing And Verification

Minimum automated checks:

- `pnpm generate:importmap`
- `pnpm generate:types`
- `pnpm lint`
- `pnpm build`

Manual media happy path:

1. Configure `BLOB_READ_WRITE_TOKEN` and `NEXT_PUBLIC_BLOB_HOSTNAME`.
2. Start MongoDB and Next.js.
3. Open `/admin`.
4. Upload a JPEG/PNG/WebP image into Media.
5. Confirm the Media document has a Blob public URL.
6. Add the image as a post `coverImage`.
7. Open `/blog` and `/blog/[slug]`.
8. Confirm the cover image renders and browser network requests go to Vercel Blob.
9. Try uploading a non-image file and confirm it is rejected.

## Acceptance Criteria

This slice is done when:

1. Payload media uploads use Vercel Blob when `BLOB_READ_WRITE_TOKEN` is configured.
2. Vercel production cannot silently fall back to local file uploads.
3. `.env.example` documents Blob variables and contains no credential-like stray line.
4. README explains how to configure Vercel Blob for local and production usage.
5. Media uploads are limited to expected image MIME types.
6. Post cover images render on `/blog`.
7. Post cover images render on `/blog/[slug]`.
8. Post Open Graph metadata includes the cover image when available.
9. `next/image` is configured for the exact Blob hostname.
10. `pnpm lint` and `pnpm build` pass.
